# 头绪工单系统项目指南

> 更新时间：2026-05-02
> 当前策略：先完善工单系统底座，再承接业务流程

## 1. 项目当前定位

头绪工单系统 当前是一个“可继续开发的权限管理基础框架”，核心目标不是继续铺功能面，而是先把已经写出来的模块稳定下来，确保登录、权限、用户、字典、文件、日志这些基础能力可持续使用。

当前不再把“代码生成器”作为第一优先级。`tool/gen` 入口和后端目录暂时保留，但在已完成模块没有稳定前，不继续扩展该功能。

面向后续“毕设管理系统”这类业务项目，头绪工单系统 的目标应先调整为：登录、权限、基础字典、文件、日志、数据权限、用户端接入都稳定后，再开始写客户、订单、支付、交付等业务模块。

## 3. 本轮排查后确认的问题

### 3.1 已修复

- 本地文件静态映射路径写死
  - 原问题：上传路径依赖数据库配置，但静态资源映射固定写死在 `E:/tx_ticket_upload/`
  - 结果：只要系统配置里修改过本地存储目录，就会出现“上传成功但访问不到”的问题
  - 处理：已改为启动时读取 `local.storage.path`

- 个人中心头像上传依赖文件管理权限
  - 原问题：个人中心走的是 `/system/file/upload`
  - 结果：普通登录用户可能没有 `sys:file:upload` 权限，导致头像上传失败
  - 处理：已改为走登录可用的 `/system/upload`

- 个人中心资料读取依赖用户管理权限
  - 原问题：个人中心直接调用 `/system/user/detail/{id}`
  - 结果：该接口绑定 `sys:user:list` 权限，普通用户可能无法访问自己的资料
  - 处理：后端新增 `/system/user/profile`，前端已切换

- 前端构建被代码生成页阻断
  - 原问题：`rbac-vue/src/views/tool/gen/index.vue` 存在不稳定实现，导致 `vite build` 失败
  - 处理：已改为稳定占位页，明确暂停开发该功能

- 通用上传接口与存储策略配置不一致
  - 原问题：`/system/upload` 被硬编码到本地存储实现
  - 结果：即使系统配置切换为其他存储策略，头像等通用上传仍会落本地
  - 处理：已改为按 `system.storage.active` 动态选择存储实现

- 表结构与当前代码不一致
  - 原问题：`sys_user` 缺少 `sex`、`birth_date`、`address`，`sys_dict_type` 存在 `update_timeMap` 错列名，`sys_permission` 缺少 `remark`
  - 结果：用户资料编辑、字典映射、权限实体都会出现结构性隐患
  - 已存在旧库时，执行 [scripts/schema_upgrade_all.sql](/E:/code/gemini/个人工单系统/scripts/schema_upgrade_all.sql)

- 按钮权限模型无法支撑跨菜单复用
  - 原问题：旧模型把按钮权限直接挂在 `sys_permission.menu_id` 上，同一权限难以在多个菜单复用
  - 结果：用户看到菜单 B 时，无法优雅复用菜单 A 上已定义过的同一按钮权限
  
- 文件上传成功但无法回显、预览、下载
  - 原问题：预览和下载接口被登录拦截，图片标签与普通链接无法携带认证头；部分本地存储配置还把预览接口地址写入 `sys_file.url`，导致后端反解磁盘路径失败
  - 处理：文件预览/下载接口按 ID 放行访问，本地文件入库地址稳定为 `/tx_files/yyyy/MM/dd/file`，并兼容旧的 `/system/file/preview/yyyy/...` 数据
  - 前端：头像改为正式文件引用，附件管理增加图片缩略图、下载流处理和当前页预览弹窗

- 文件统计报表在 MySQL `ONLY_FULL_GROUP_BY` 下报错
  - 原问题：趋势 SQL `SELECT DATE_FORMAT(create_time)` 与 `GROUP BY DATE(create_time)` 表达式不一致
  - 处理：统一为 `GROUP BY DATE_FORMAT(create_time, '%Y-%m-%d')`

- 文件预览能力不足
  - 处理：新增 `rbac-vue/src/components/file-preview`，支持图片、PDF、docx、xlsx/xls、pptx 预览，不支持的老格式提示下载
  - 约束：`.doc`、`.ppt` 这类老 Office 格式暂不做浏览器端渲染，后续如必须支持应考虑服务端转换

- 菜单、角色、部门树形数据检索复用不足
  - 处理：前端新增树搜索工具组件，后端新增通用树构建工具，后续树形查询优先复用该能力

### 3.2 已发现但需持续跟进

- Office 预览包体较大
  - 当前处理：文件预览组件使用异步组件加载，避免在进入系统时一次性加载所有 Office 预览代码
  - 后续建议：如文件预览成为高频能力，可以继续按文件类型拆分更细的懒加载策略

- 代码生成器后端接口仍保留
  - 当前处理方式：前端先停用为占位说明，不继续扩展
  - 后续建议：等已完成功能稳定后，再决定保留、重写或废弃

- UniApp 当前仍需单独回归
  - 用户、角色、字典、文件等页面曾出现接口路径、字段名、权限标识与后端不一致的问题
  - 文件预览、下载、上传需要按文件 ID 和真实后端接口重新验证
  - 用户端接口后续应与管理端接口分离，避免客户越权访问后台业务数据
  - 修复 UniApp 前应重新静态检查当前代码，不再依赖单独的临时 Bug 文档

## 4. 当前推荐开发顺序

1. 完成 RBAC 核心闭环：登录、菜单、按钮权限、角色授权、用户直授、权限缓存刷新
2. 完成基础管理回归：用户、角色、菜单、部门、字典、文件、日志、Dashboard
3. 补齐文件能力回归：上传、头像、预览、下载、按 ID 引用、Office 预览
4. 补齐接口规范：分页、DTO/VO、异常返回、导出、权限注解、操作日志
5. 评估并修复 UniApp，至少保证登录、token、当前用户数据隔离、文件预览下载稳定
6. 补齐数据权限能力，支持未来按负责人、部门、角色隔离业务数据
7. 清理初始化脚本和升级脚本，保证新项目能直接初始化数据库
8. 最后再考虑恢复代码生成器或开发毕设管理系统业务模块

## 5. 文件模块当前约束

- 本地存储依赖 `local.storage.path`
- 本地访问路径默认使用 `/tx_files/**`
- 前端业务侧优先使用文件 ID 访问 `/system/file/preview/{id}` 与 `/system/file/download/{id}`
- 文件管理页上传会写入 `sys_file`
- 通用上传接口主要用于头像等轻量场景，当前也会写入 `sys_file`，便于统一回显、预览和未来文件服务解耦
- Web 端统一使用 `FilePreview` 组件处理文件预览，业务页面不要重复拼接本地存储路径

## 5.1 数据库迁移说明

- 新环境直接执行 [init.sql](/E:/code/gemini/头绪工单系统/init.sql)
- 已存在旧库时，执行 [scripts/schema_upgrade_all.sql](/E:/code/gemini/个人工单系统/scripts/schema_upgrade_all.sql)
- 当前环境没有 MySQL 客户端，因此本轮只提交了迁移脚本，未在本机直接执行

## 5.2 当前权限模型

- `sys_menu`
  - 只负责页面、路由、菜单可见性
- `sys_permission`
  - 只负责全局按钮权限定义
- `sys_menu_permission`
  - 负责菜单挂载哪些权限
- `sys_role_perm`
  - 负责角色拥有的按钮权限
- `sys_user_perm`
  - 预留用户直授权能力

## 5.3 大规模权限下的处理策略

- 后端不再按菜单逐次查按钮，而是汇总用户有效权限并缓存到登录会话
- 角色授权页的按钮权限改为分页查询，避免一次把大量权限全部压到前端
- 前端 `v-hasPerm` 改为内存 `Set` 判断，避免每次渲染反复 `JSON.parse`

## 6.1 面向业务系统前必须补齐的 RBAC 能力

在开始毕设管理系统、订单系统、客户系统之前，头绪工单系统 至少要达到以下标准：

- 权限闭环稳定：菜单权限、按钮权限、角色授权、用户直授、前端权限指令、权限缓存刷新全部可用
- 文件闭环稳定：业务只保存文件 ID，预览和下载统一走后端接口，前端不拼接物理路径
- 数据权限可扩展：后续业务订单能按负责人、部门、角色范围隔离数据
- 用户端隔离明确：管理端接口和用户端接口分离，用户端只能访问自己的数据
- 字典可复用：业务状态、支付状态、文件类型、客户来源等都能通过字典维护
- 日志可追溯：关键后台操作、登录行为、文件操作都有记录
- 数据库脚本干净：`init.sql` 与升级脚本一致，避免新环境手工补字段

## 6.2 毕设管理系统承接原则

毕设管理系统不应改造 `sys_user`、`sys_role`、`sys_dept` 来承载客户、订单、支付、交付资料。

推荐做法：

- RBAC 只负责账号、后台权限、菜单按钮、基础系统能力
- 客户档案、选题、订单、支付、交付资料、AI 问答记录放到独立业务模块
- 后端新增业务包建议放到 `modules.biz`
- 支付能力独立为 `modules.payment`
- AI 选题能力独立为 `modules.ai`
- 用户端接口使用独立前缀，例如 `/portal/**`，不要直接暴露管理端 `/biz/**` 接口

## 7. 开发原则

- 先修已完成模块，不抢新功能
- 接口路径、权限、返回结构必须统一
- 配置驱动的能力不能再写死路径或策略
- 每次变更都保持最小改动，避免引入新的结构性风险
- 新业务代码必须按领域进 `modules.biz`，禁止直接平铺到根包
- 统计、报表、导出 SQL 必须放在 Mapper，不要写在 Service

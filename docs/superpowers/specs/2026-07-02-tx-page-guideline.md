# 头绪工单系统页面规范

## 1. 目标

本规范用于统一 `头绪工单系统` 的页面结构、视觉风格、交互方式与查询设计，避免页面风格漂移、信息层级混乱、查询条件失控。

目标只有四个：

1. 企业后台风格统一。
2. 页面以业务内容为中心，不做营销页式堆叠。
3. 常规列表页、表单页、导航页、报表页各有固定模板。
4. 查询设计必须兼顾后续百万级数据量，不允许无节制的多字段 `or like`。

---

## 2. 总体设计原则

### 2.1 总体视觉方向

- 使用企业级中后台风格，不做花哨落地页。
- 主体以浅色背景、白色卡片、深灰文本为主。
- 主色优先沿用系统默认主色，局部允许使用默认主色渐变。
- 渐变、动画只允许用于 `导航页`、`报表页`、少量关键入口卡片，不允许滥用于表格页和表单页。
- 页面首屏应优先出现业务内容，不允许被大段说明文案抢占。

### 2.2 页面层级原则

- 第一视觉层：页面标题、主操作、关键状态。
- 第二视觉层：筛选区、统计区、辅助入口。
- 第三视觉层：说明文案、帮助提示、次要说明。

### 2.3 卡片与容器原则

- 默认使用白色卡片承载业务模块。
- 卡片圆角、阴影、边框统一，不同页面不得各自定义一套。
- 同一页面卡片宽高节奏应统一，避免有的很高、有的很扁。

### 2.4 颜色原则

- 主按钮使用系统主色。
- 次按钮使用默认描边或浅底。
- 危险操作只用红色。
- 表格页不使用大面积高饱和背景。
- 除导航页、报表页外，禁止使用大面积渐变横幅。

### 2.5 动效原则

- 动效只用于增强层次，不用于制造存在感。
- 导航页可使用轻量进入动画、数字过渡、卡片悬停。
- 表格页、表单页只允许轻微过渡，不允许大幅位移动画。
- 动效时长保持简短，避免影响后台操作效率。
- 导航页允许保留轻量渐变和动画，但必须服务于信息层级，不能喧宾夺主。

### 2.6 布局与密度原则

- 页面主体宽度保持稳定，优先使用统一内容容器，不同页面不要忽宽忽窄。
- 卡片间距、表单行距、表格上下留白应统一，避免某些页面“很松”、某些页面“很挤”。
- Table 页优先高密度布局，导航页与报表页可以适当拉开层次。
- 表单页优先保证信息可读性，其次再考虑装饰性。
- 同一业务模块内，按钮高度、输入框高度、卡片标题样式必须统一。

## 3. 页面分类规范

## 3.1 导航页规范

适用场景：

- 工单首页
- 模块首页
- 工作台首页

设计要求：

- 导航页允许内容相对丰富，但必须简洁有层次。
- 可以使用默认主色渐变、数据卡片、轻量动画、快捷入口卡片。
- 首屏必须有明确主操作，如 `提交工单`、`进入工作台`。
- 首屏必须有真实业务内容，不能只有标题和介绍。
- 快捷入口可以采用卡片式，但数量要克制，避免堆满。
- 文案只保留一句核心说明，不写大段介绍。

推荐结构：

1. 页头区：页面标题 + 主按钮 + 次按钮
2. 核心业务区：最近工单 / 待办 / 我的任务 / 重要统计
3. 辅助入口区：快捷入口、知识库入口、系统配置入口
4. 扩展区：趋势、公告、常用报表

限制：

- 不允许做成营销官网风格。
- 不允许标题区高度过高。
- 不允许页头把首屏业务内容顶到下方。

---

## 3.2 Table 页面规范

适用场景：

- 工单工作台
- 我的工单
- 知识库列表
- 文件管理
- 系统配置列表

设计要求：

- Table 页面必须简洁、紧凑、可高频操作。
- 页面最上方不做大提示横幅。
- 标题区只保留 `页面标题 + 右上角主操作按钮组`。
- 查询条件统一放在表格上方，且默认支持折叠/展开。
- 筛选区默认展开，确保用户打开页面后立即可见常用查询项。

按钮布局规则：

- `新增`、`导出`、`导入`、`批量操作` 统一放在表格右上角。
- `查询`、`重置` 放在筛选区内。
- 行内按钮按类型排布：先业务主动作，再辅助动作，危险动作放最后。

查询区规则：

- 默认展示常用条件，且首屏默认展开。
- 高级条件进入折叠区。
- 筛选字段按业务拆分，不允许一个输入框承载所有模糊查询。
- 日期范围、状态、优先级、分类等应优先用结构化筛选。
- Table 页统一强制启用筛选折叠，保持页面交互一致。
- 筛选布局优先参考 `system/file` 页的紧凑 inline 结构，再按业务字段数量做换行。

推荐结构：

1. 页头：标题 + 右上角主操作按钮组
2. 查询卡：常用条件 + 折叠高级筛选
3. 表格卡：表格主体 + 表头工具区 + 分页
4. 扩展区：必要时放统计、说明、批量结果，不默认常驻首屏

表格规则：

- 列字段按业务优先级排序，最重要字段靠左。
- 操作列固定在右侧。
- 时间统一显示 `YYYY-MM-DD HH:mm:ss`。
- 状态、优先级统一使用标签色表示。
- 行操作文案统一短词，例如 `详情`、`处理`、`编辑`、`导出`。

查询设计规则：

- 禁止前端长期依赖单个 `keyword` 覆盖所有字段。
- 列表查询优先使用独立字段：
  - 编号
  - 标题
  - 分类
  - 状态
  - 优先级
  - 客户
  - 处理人
  - 时间范围
- 后端禁止对多个大字段长期使用 `or like` 全模糊查询。
- 对高频筛选字段必须建立索引。
- 字符串查询优先使用 `=` 或前缀匹配，少用 `%keyword%` 全模糊。
- `keyword` 仅保留为兼容兜底能力，不再作为后台列表页默认主查询方式。
- `keyword` 更适合 App 端、全局搜索、轻量入口等场景，后台 PC 页优先结构化筛选。

参考方向：

- `system/file` 页的整体节奏可以参考。
- 在其基础上补充 `查询区折叠能力`。

## 3.3 新增 / 修改页面规范

适用场景：

- 创建工单
- 编辑知识库
- 新增系统配置
- 编辑规则项

选择方式规则：

- 内容较少：使用 `Dialog` 或 `Drawer`
- 内容较多：使用独立页面跳转
- 涉及多段说明、附件、流程、AI、复杂表单时，必须使用独立页面

强制规则：

- 独立页面必须有清晰的返回按钮。
- 页面结构必须均匀，不允许一边很挤一边很空。
- 表单字段必须分组展示，不允许无节制地从上到下堆字段。
- 附件、图片、视频等辅助内容，应放到侧栏或下方卡片，不抢主表单。
- AI 辅助区不固定位置，应根据页面主任务决定摆放方式。
- 当 AI 为当前页面的高频入口时，可放在顶部或做成悬浮入口。
- 当 AI 仅为辅助能力时，可放在侧栏、局部卡片或“系统精灵”式入口中。
- AI 区域必须保持轻量，不允许挤压主业务表单首屏。
- 提交、保存类主按钮优先统一放在页面或表单右下角，不与页头按钮重复出现。

推荐结构：

1. 页头：返回按钮 + 页面标题 + 主提交按钮
2. 辅助区：AI、提示、说明
3. 主表单区：核心业务字段
4. 附件区：图片、文档、视频
5. 底部操作区：取消 / 保存 / 提交

推荐布局：

- 双栏场景：左侧主表单，右侧辅助信息卡 / 提单说明 / 附件预览
- 单栏场景：上方 AI 与说明，中部主表单，下方附件与操作区
- 复杂流程页：顶部摘要，左侧客户原始信息，右侧处理记录与流转动作

表单规则：

- 必填项必须清晰。
- 长文本区域优先给足高度。
- 同类字段横向对齐。
- 不允许按钮大小不一致。
- 不允许主次按钮混乱。

## 3.4 报表展示页面规范

适用场景：

- 工单统计看板
- 文件统计报表
- 知识库沉淀分析
- 后续 Agent 数据分析页

设计方向：

- 报表页允许比普通表格页更有设计感。
- 可以适度使用渐变、数字动效、图表动画、卡片层级。
- 重点是“数据解读”，不是“装饰”。
- 报表页允许单独采用稍强的视觉表达，但不得破坏阅读效率与信息优先级。

推荐结构：

1. 顶部概览：核心指标卡
2. 中部趋势：时间趋势图、分布图、排行图
3. 下部明细：明细表、异常项、 drill-down
4. 侧边或下方：说明、口径、洞察结论

报表规则：

- 每页必须有 1 组核心指标卡。
- 图表必须有标题，必要时有副标题或口径说明。
- 图表类型优先选择易读类型：
  - 趋势：折线图
  - 对比：柱状图
  - 构成：饼图 / 环图
  - 排行：横向条形图
- 报表页可以加入适度动画，但不能影响数据阅读。
- 报表页也不要使用长篇说明横幅。

我的补充建议：

- 报表页面应尽量提供 `筛选 -> 指标卡 -> 趋势 -> 明细` 的固定顺序。
- 如果后续要接 Agent，图表下方应保留结构化字段名称，方便二次分析。
- 报表结论最好能沉淀成“可读洞察”，而不是纯图表堆砌。

## 4. 查询与性能规范

该部分为强制规则。

### 4.1 前端查询规范

- 一个筛选项只表达一个业务含义。
- 不允许把编号、标题、描述、客户、处理人长期混在一个 `keyword` 输入框里。
- 对高频条件必须结构化展示。
- 高级查询放入折叠区，默认折叠。

### 4.2 后端查询规范

- 禁止默认使用多个字段 `or like` 拼接做主查询逻辑。
- 对编号、状态、优先级、分类、用户、时间等字段优先使用索引友好条件。
- 标题、名称类字段优先使用前缀模糊，而不是全字段 `%keyword%`。
- 对 text / 大字段禁止作为常规列表页主查询条件。

### 4.3 索引规范

高频列表页必须至少保证：

- 编号索引
- 状态索引
- 优先级索引
- 分类索引
- 创建人 / 处理人索引
- 创建时间索引

### 4.4 百万级数据预留规范

- 列表页必须默认分页。
- 不允许一次拉全量。
- 查询条件必须可组合且可走索引。
- 导出必须单独控制，不可直接复用列表无约束条件。

## 5. 通用交互规范

### 5.1 标题与按钮

- 页面标题保持统一大小和位置。
- 主按钮放右侧。
- 次按钮紧随主按钮。
- 危险按钮不与保存按钮并列混放。

### 5.2 文案

- 文案短、准、业务化。
- 禁止口号式描述。
- 禁止页面首屏大段解释说明。

### 5.3 空状态

- 空状态要说明“当前没有什么”。
- 空状态优先提供下一步动作入口。

### 5.4 返回机制

- 所有独立表单页必须提供返回。
- 处理页、详情页、创建页不允许无返回入口。

### 5.5 响应式规则

- PC 端为主，但必须兼容常见笔记本分辨率，不允许出现首屏核心表单被挤压到不可用。
- 当屏幕变窄时，双栏结构应自动降级为上下结构。
- Table 页在窄宽度下优先保证核心列可读，其次再折叠次要列或转入详情查看。
- 表单页在窄宽度下禁止按钮换行错位、字段宽度忽长忽短。

## 6. 明确禁止

以下做法禁止继续出现：

- 大面积绿色或渐变横幅出现在普通 table 页或表单页顶部
- 首屏长说明文案挤压业务内容
- 一个输入框承担所有查询字段
- 行内按钮顺序混乱
- 主次按钮大小不一致
- 表单左右密度失衡
- 附件区、AI 区抢占主业务区
- 同类页面风格完全不一致

## 7. 当前项目落地建议

优先按以下顺序整改：

1. `提交工单`
2. `工单工作台`
3. `我的工单`
4. `工单首页`
5. `处理工单`
6. `知识库`
7. `系统配置`

推荐统一基座：

- 页头统一短标题栏
- 查询区统一折叠筛选卡
- 表格区统一右上角按钮布局
- 表单页统一返回按钮和底部操作区
- 报表页单独允许更强视觉表现

建议补充的页面基座组件：

- `PageHeader`：统一标题、返回、主按钮、次按钮
- `FilterPanel`：统一查询区折叠、展开、重置、查询
- `ActionToolbar`：统一新增、导出、导入、批量按钮布局
- `InfoCard`：统一统计卡、摘要卡、状态卡样式
- `PageSection`：统一卡片分区标题、间距、内容容器

## 8. 已确认结论

以下内容已确认，后续按此执行：

1. `导航页` 允许保留轻量渐变和动画，但仅限增强层级与质感。
2. `AI 辅助区` 不固定在顶部，应根据页面任务决定位置，必要时可做成系统精灵式入口。
3. `Table 页` 统一强制启用筛选折叠，保持交互一致。
4. `keyword` 只保留兼容兜底用途，不再作为后台列表默认主查询方式，主要服务 App 端等轻入口场景。
5. `报表页` 允许单独采用稍强视觉表达，但必须坚持数据优先。

## 9. 本轮补充约束

以下规则自 2026-07-02 起作为工单中心强制补充规范：

1. `我的工单` 明确为客户视角页面，只承载草稿、已提交、已关闭与详情查看，不放运维接单、分派、转派动作。
2. `待接工单` 必须独立成单独 table 页；`工单工作台` 仅保留 `我处理中 / 我已完成 / 全部工单`，且切换 tab 必须放在 table card 内、紧贴表格上方，不允许再挂在页面最上方。
3. `系统配置` 必须采用标准 table 页结构：筛选区 + 表格区 + 弹窗编辑；禁止再做左右分割编辑页。
4. `提单页` 的 `附件与素材` 必须使用正常按钮入口，不允许再出现大面积上传占位块；底部 `取消 / 草稿 / 提交` 尺寸必须一致。
5. `流程页 / 详情页` 的留痕展示优先采用论坛式、自上而下的节点流，每个节点必须明确动作、人、时间、内容，不能只剩一条弱化的时间线。
6. 工单模块内的状态、优先级、问题类型、系统配置统一从后端字典接口获取；前端不得再长期依赖本地全局字典常量作为主数据源。

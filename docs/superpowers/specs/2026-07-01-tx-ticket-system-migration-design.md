# 头绪工单系统迁移与命名改造设计

## 1. 背景与目标

- 迁移源项目：`E:\code\gemini\RBAC-BASE`
- 迁移目标目录：`E:\code\gemini\个人工单系统`
- 本次仅迁移 Web 前端与 Java 后端，不迁移 `rbac-uniapp`
- 迁移后系统中文名称统一为 `头绪工单系统`
- 迁移后英文标识统一采用 `tx` 缩写体系，按“工单系统”语义命名

本次工作的核心目标是基于现有 RBAC 基座，快速落成一个可继续演进的工单系统项目骨架，同时完成首轮品牌化命名替换，避免保留明显的 `RBAC-BASE` 对外痕迹。

## 2. 迁移结构

目标目录保持前后端分离结构，减少改造成本并保留后续与源项目比对、补丁迁移的便利性：

- `E:\code\gemini\个人工单系统\rbac-java`
- `E:\code\gemini\个人工单系统\rbac-vue`
- `E:\code\gemini\个人工单系统\init.sql`
- `E:\code\gemini\个人工单系统\scripts\` 下必要 SQL/升级脚本
- `E:\code\gemini\个人工单系统\README.md`、`PROJECT_GUIDE.md` 等必要文档

## 3. 文件迁移边界

### 3.1 允许迁移

- 后端源码、资源文件、Maven 配置
- 前端源码、公共资源、Vite 配置、依赖声明
- 数据库初始化脚本与必要升级脚本
- 项目说明文档

### 3.2 不迁移

- `rbac-uniapp`
- `node_modules`
- `dist`
- `target`
- `.idea`
- 运行日志
- 其他缓存与中间产物

## 4. 命名改造范围

### 4.1 中文名称

对外展示名称统一改为：

- `头绪工单系统`

影响范围包括但不限于：

- 浏览器标题
- 登录页主标题、副标题
- 布局 Logo 文案
- 文档标题
- 后端启动提示文案

### 4.2 英文名称与缩写

本次统一采用 `tx` 作为缩写前缀，推荐标识如下：

- 前端工程名：`tx-ticket-web`
- 后端工程名：`tx-ticket-server`
- 英文系统名：`TX Ticket System`
- 数据库名：`tx_ticket`
- 本地存储目录默认值：`E:/tx_ticket_upload/`
- 本地文件访问前缀：`/tx_files/`
- 统计导出文件名中的 `rbac` 前缀改为 `tx-ticket`

## 5. 技术边界与风险控制

### 5.1 本次明确要改的标识

- `pom.xml` 中 `artifactId`、`name`、`description`
- 前端 `package.json` 的 `name`
- `index.html` 标题
- 登录页与布局页展示文案
- 数据库初始化脚本中的库名、默认存储路径、默认访问前缀
- 后端默认 JDBC 库名配置
- 前端/后端中直接写死的 `rbac_upload`、`rbac_files`、`rbac-analytics` 等外显标识

### 5.2 本次不做的重构

为控制改造风险，本次不执行以下高成本重构：

- 不重命名 Java 包路径 `com.rbac.base`
- 不批量改动现有 `/rbac/...` 接口路径
- 不调整现有菜单、权限码、模块目录 `rbac`
- 不将系统直接改造成完整“工单业务”功能，仅完成系统底座迁移与首轮品牌命名替换

这样处理的原因是：若连包路径、接口路径、权限路径一起替换，会从“迁移 + 改名”升级为“整库重构”，显著扩大改动面并增加启动失败风险。

## 6. 实施顺序

1. 从源目录复制后端、前端、SQL、必要脚本和文档到目标目录
2. 过滤构建产物、日志与 IDE 目录
3. 在目标目录内执行首轮命名替换
4. 核对目录完整性与关键配置是否一致
5. 视环境情况执行基础启动级校验

## 7. 验收标准

- 目标目录具备完整的 `rbac-java` 与 `rbac-vue` 两套工程
- 不包含 `rbac-uniapp` 与无关构建垃圾
- 用户可在界面和项目配置中看到 `头绪工单系统` / `TX Ticket System` 相关命名
- 默认数据库名、文件存储路径、文件访问前缀已切换为 `tx` 体系
- 保留原有代码结构与主要接口路径，确保后续继续开发风险可控

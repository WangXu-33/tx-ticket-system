# 头绪工单系统业务规格

## 背景

头绪工单系统（TX Ticket System）面向客户问题受理、运维处理、转派协同、流程留痕和知识沉淀。现有项目已经具备 RBAC、用户、角色、菜单、文件上传、通知公告、实时连接等基础能力，新业务应作为独立工单模块接入，避免污染权限基础模块。

## 目标

1. 客户可以注册账号并提交工单。
2. 运维人员可以接单、处理、回复、转派、解决工单。
3. 系统完整保留工单生命周期流程记录，包括处理人、时间、动作、原因、附件和客户可见范围。
4. 支持图片和附件上传，附件可以绑定到工单、回复、流程记录或知识库文章。
5. 建立知识库，用已解决工单沉淀标准解决方案，为后续 Agent 推荐方案预留结构化数据。

## 角色

| 角色 | 英文标识 | 说明 |
| --- | --- | --- |
| 客户 | customer | 注册后提交和跟踪自己的工单，补充资料，确认关闭，评价结果 |
| 一线客服 | support | 受理工单、初步处理、回复客户、转派运维 |
| 运维人员 | operator | 处理技术问题、补充处理记录、提交解决方案 |
| 专家 | expert | 处理升级或复杂问题，维护知识库 |
| 主管 | supervisor | 查看团队工单、审批知识库发布、处理超时和统计报表 |
| 管理员 | admin | 管理用户、角色、菜单、字典、系统参数 |

## 工单状态机

| 状态 | 编码 | 说明 |
| --- | --- | --- |
| 待受理 | pending | 客户提交后尚未分配或接单 |
| 处理中 | processing | 已分配处理人，正在处理 |
| 待客户补充 | waiting_customer | 运维需要客户补充信息 |
| 已转派 | transferred | 工单发生转派，当前处理人已变更 |
| 已解决 | resolved | 运维给出解决方案，等待客户确认 |
| 已关闭 | closed | 客户确认或系统自动关闭 |
| 已驳回 | rejected | 无效、重复或不属于服务范围 |

状态流转必须写入流程记录，不允许只改主表状态。

## 核心流程

### 客户提单

1. 客户注册账号。
2. 客户填写标题、问题描述、分类、优先级、联系方式、附件。
3. 系统生成工单编号，默认状态为 `pending`。
4. 系统写入一条 `created` 流程记录。

### 运维处理

1. 运维从工单池接单或由主管分派。
2. 处理人回复客户或添加内部备注。
3. 需要客户补充时，状态改为 `waiting_customer`。
4. 处理不了时，填写转派原因并选择新处理人。
5. 给出解决方案后，状态改为 `resolved`。
6. 客户确认后关闭；长期未确认可由系统或主管关闭。

### 知识沉淀

1. 已解决工单可以生成知识库草稿。
2. 草稿包含问题现象、原因分析、解决步骤、适用范围、标签、关联工单。
3. 主管或专家审核后发布。
4. 后续 Agent 根据标题、问题描述、标签、分类、解决步骤做检索和推荐。

## 数据范围

1. 客户只能查看自己创建的工单。
2. 运维可以查看分配给自己的工单、自己处理过的工单、权限范围内的团队工单。
3. 主管可以查看团队工单和统计数据。
4. 管理员可以查看所有工单。

## 附件规则

1. 附件统一复用现有 `sys_file` 上传能力。
2. 业务关联表只保存 `file_id`、业务类型、业务 ID、上传人、可见范围。
3. 图片和普通附件均作为文件记录管理，业务页面根据文件后缀做预览或下载。

## 知识库 Agent 预留字段

知识库必须结构化保存以下内容：

1. 问题现象：客户问题的标准化描述。
2. 原因分析：根因、环境、触发条件。
3. 解决步骤：可执行的操作步骤。
4. 适用范围：产品、系统、版本、客户类型。
5. 标签关键词：用于全文检索和向量检索。
6. 关联工单：知识来源和效果追溯。
7. 命中反馈：客户是否认为有帮助。

## MVP 范围

1. 客户注册接口。
2. 工单列表、详情、创建、回复、接单、分派、转派、解决、关闭、驳回。
3. 工单流程记录。
4. 工单附件关联。
5. 知识库文章列表、详情、新增、编辑、发布、下架、关联工单。
6. 前端菜单和基础页面。
7. 初始化 SQL 包含业务表、角色、菜单、权限、字典。

## 暂不纳入 MVP

1. Agent 自动回答。
2. 向量数据库。
3. 短信、邮件通知。
4. SLA 自动升级。
5. 复杂排班和值班规则。
6. 多租户计费。
